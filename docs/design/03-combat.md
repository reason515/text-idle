# 战斗系统设计详细说明

## 一、回合与出手顺序 (Turn Order)

### 1.1 出手顺序规则

战斗采用**回合制**，每回合内所有参战单位（玩家英雄与怪物）按**敏捷 (Agility)** 从高到低依次行动：

| 规则 | 说明 |
|------|------|
| **敏捷决定顺序** | 敏捷越高的角色（含怪物）越先出手 |
| **同敏捷处理** | 敏捷相同时，出手顺序随机决定 |
| **透明化** | 战斗日志中记录出手顺序与行动结果，便于战后复盘与战术调整（本游戏为放置游戏，战斗过程玩家无法实时操控，仅能通过预先配置的战术策略间接影响） |

- **设计意图**：敏捷不仅影响暴击、闪避，还直接影响行动先后，使高敏捷职业（盗贼、猎人、德鲁伊）在节奏上有优势。

#### 1.1.1 第一回合「开怪」先手（Pull opener）

仅**第 1 回合**的第一动特殊，用于模拟魔兽式「开怪」、便于坦克先建立仇恨：

| 条件 | 第一动 |
|------|--------|
| 小队已**指定坦克**且该英雄本场存活 | **指定坦克**先行动 |
| **未指定坦克** | 在**存活我方英雄**中**随机**一人先行动 |
| 从**第二动起**（含第 1 回合内后续单位） | 与第 2 回合起相同：敌我混合，**仅按敏捷**从高到低排序（同敏捷随机） |

- **设计意图**：避免坦克敏捷偏低时，高敏输出/治疗在第 1 回合抢先动手导致过早 OT；未指定坦克时仍由随机我方成员开怪，保持规则简单。

### 1.2 每回合行动次数

| 规则 | 说明 |
|------|------|
| **每回合每角色行动一次** | 每个参战单位（英雄或怪物）每回合仅能执行一次行动（普攻或技能） |
| **行动类型** | 行动可为普攻、主动技能或待机等，由战斗 AI 或玩家策略决定 |

- **小数值与节奏**：单回合单行动保证战斗节奏清晰，便于仇恨、治疗与打断的时序规划。

### 1.3 客户端战斗日志播放节奏（仅表现层）

| 项目 | 说明 |
|------|------|
| **含义** | 主界面战斗日志**逐条播放**时，相邻两条之间的等待时间（毫秒），便于观察单次行动与回合结束后的状态；**不改变**服务端或本地战斗结算的回合规则与数值。 |
| **默认** | 每条日志步骤间隔 **3000ms**（含「行动前」与「回合分隔后」两处与步骤相关的停顿，与实现一致）。 |
| **可调** | 代码常量 `DEFAULT_COMBAT_LOG_STEP_DELAY_MS`；构建时可设环境变量 `VITE_COMBAT_LOG_STEP_DELAY_MS`；运行时可在浏览器 `localStorage` 设置键 `textIdleCombatLogStepDelayMs`（非负整数，优先级高于 Vite 环境变量）。 |
| **与休息步一致** | 战后休息**每一步**在日志中的 reveal 间隔，须与**本条**「每条日志步骤间隔」使用**同一毫秒值**（单一配置源），使统计上的**战斗行动步**与**休息步**在观感上等长；设计依据见 [13-player-statistics.md](./13-player-statistics.md) 7.5。实现上 `COMBAT_PACING_MS.restStepReveal` 应与 `getCombatLogStepDelayMs()` 对齐（或其一派生自另一）。 |
| **其他正式服节奏** | 地图切换分隔、地图描述展示、遭遇信息、战败后间隔、战后循环间隔等毫秒数集中在 `frontend/src/game/combatPacing.js` 的 `COMBAT_PACING_MS`（与回合结算逻辑无关，仅 UI 表现）。 |
| **E2E** | 自动化测试通过 `localStorage` 键 `e2eFastCombat=1` 或 URL 查询 `?e2e=1` 进入快速模式；`applyCombatPacingDelayMs` 对上述正式服毫秒数与日志步骤间隔一律返回 **0ms**，与玩家正式运行时的节奏**明确区分**（不依赖 `navigator.webdriver` 等隐式检测）。**例外**：战败结算小结出现后、进入战后休息前的真实停顿长度由 **`getDefeatBeforeRestPauseMs()`** 单独供给（正式服仍等价于 `COMBAT_PACING_MS.defeatBeforeRest`；快速模式下为约 **520ms**），避免因 `autoRest` 瞬时清空遭遇/回血导致界面留不住「阵亡态」而产生自动化竞态或难以人工观察的细节帧。 |

## 二、怪物数量与战斗规模 (Encounter Size)

### 2.1 怪物数量分布

单场战斗中怪物数量与玩家小队规模的关系如下：

| 情况 | 概率 | 说明 |
|------|------|------|
| **与队伍人数相当** | 较大概率 | 如 3 人队伍时，多数场次为 3 只怪 |
| **少于队伍人数** | 较小概率 | 如 3 人队伍时，少数场次为 1–2 只怪，节奏更快 |
| **多于队伍人数** | 较小概率 | 如 3 人队伍时，少数场次为 4–5 只怪，难度提升 |

- **可配置**：具体概率分布（如 70% 相等、15% 少、15% 多）可通过配置调节，便于平衡调试。
- **设计意图**：多数场次势均力敌，少数场次制造节奏变化与难度起伏。

## 三、胜负判定 (Victory & Defeat)

| 结果 | 触发条件 | 后续 |
|------|----------|------|
| **胜利** | 怪物全部死亡 | 进入战后结算（经验、金币、掉落）；进入休息恢复阶段（自动执行） |
| **失败** | 玩家英雄全部死亡 | 战斗结束；不获得本场掉落；**扣除当前地图探索度 10 点（可配置），最低不低于 0**；短暂等待后自动开始下一场战斗 |

- **战败惩罚**：每次战败扣除探索度（默认 10 点，可配置），防止零成本无限刷怪；探索度不会因战败归零，保留已积累的进度。
- **透明化**：胜负判定逻辑对玩家完全公开，便于评估风险与策略取舍。**战斗日志每条战报的结算小节**中会写出本场**探索进度**的增加量、战败扣减量（数值为面板上的实际增减，会因已达 0 / 已满 100 而与「尝试扣除/理论击杀累加」不同），以及击败守关 BOSS 后「进入下一张地图、进度重置」的提示（与 [02-levels-monsters.md](./02-levels-monsters.md) 探索度小节一致）。

## 四、战后休息与恢复 (Post-Combat Rest)

### 4.1 恢复机制

每场战斗胜利后，队伍自动进入**休息阶段**，无需玩家手动操作，系统逐步恢复所有角色资源，直至完全恢复后自动开始下一场战斗：

| 规则 | 说明 |
|------|------|
| **自动执行** | 休息恢复全程自动进行，玩家无需点击任何按钮 |
| **恢复门槛** | 所有英雄 HP 与 MP（或怒气/能量/集中值等）均恢复至满值后，自动开始下一场战斗 |
| **恢复速度** | HP、MP 的恢复速度受**精神 (Spirit)** 属性以及**装备**等因素影响 |
| **死亡惩罚** | 若有角色在本场战斗中死亡，恢复时间将延长（如恢复速度降低或需额外恢复回合），以体现战斗代价 |
| **时间单位** | 恢复按**回合数**或**时间步**结算（遵循游戏「回合制」设计，不使用秒/分钟等实时单位） |

### 4.2 恢复公式（草案）

| 资源 | 恢复公式（每回合/每步） | 主要影响因素 |
|------|--------------------------|--------------|
| HP | `HP_Regen = Base + Spi * k_HP_Regen + 装备恢复加成` | 精神、装备词缀 |
| MP | `MP_Regen = Base + Spi * k_MP_Regen + 装备恢复加成` | 精神、装备词缀 |

- **战斗内法师/牧师 MP 回复（与上表战后休息公式不同）**：每回合结束时 `floor(Spi * 0.8 + 装备恢复加成)`，详见 [05-skills.md](./05-skills.md) 8.2.1。
- **怒气/能量/集中值**：战士怒气脱战后归零，休息阶段不恢复；其他资源按职业规则处理。
- **死亡惩罚**：若有 N 名角色在本场战斗中死亡，恢复速度可乘以惩罚系数（如 `1 + N * k_DeathPenalty`），或增加固定额外恢复回合，具体数值可配置调节。
- **透明化**：恢复速度与公式对玩家可见，便于评估「精神」与恢复词缀的收益。

### 4.3 与核心循环的衔接

- **放置 (Engage)**：战斗 → 休息恢复 → 下一场战斗，形成连续的探索节奏。
- **优化 (Optimize)**：堆叠精神或恢复词缀可缩短休息时间，提高挂机效率。
- **玩家数据统计**：战斗中每个单位每次行动计 **1 战斗行动步**；每场休息所消耗的**休息步数**（每步恢复推进一次，见 4.1「时间单位」）与战斗行动步**同粒度**，二者之和为 [13-player-statistics.md](./13-player-statistics.md) 的**探索步数**，作为**金币/经验每探索步效率**分母，使死亡惩罚与长休息在效率数字上可感知。

## 五、与仇恨、伤害公式的衔接

- **仇恨机制**：怪物每回合按仇恨值选择目标，详见 [12-threat.md](./12-threat.md)。
- **伤害与减伤**：物理/魔法伤害、护甲/抗性减伤公式见 [04-classes-attributes.md](./04-classes-attributes.md) 2.2.3。
- **英雄普攻**：在 `CLASS_COEFFICIENTS` 中 **`k_PhysAtk` 为 `null` 的职业**（法师、牧师、术士等纯法系成长）普攻按**法术伤害**结算（有效法术强度、法术暴击与目标抗性）；其余职业普攻按**物理伤害**结算（有效物攻与目标护甲）。不再存在「法强高于物攻时随机打出魔法普攻」的分支。实现见 `frontend/src/game/combat.js` 中 `actorDamage`。
- **状态效果**：Buff/Debuff 对出手顺序、恢复等的影响，见「状态效果」设计。

## 六、命中与闪避判定（WoW 风格简化版）

### 6.1 设计目标

- 采用类似魔兽世界的对抗思路：**攻击方命中** 对抗 **防守方闪避**。
- 将“是否命中”置于伤害结算之前，避免出现“未命中仍触发伤害后效果”的歧义。
- 保留小数值可读性：使用百分比、上下限封顶，便于玩家理解构筑收益。

### 6.2 判定顺序（单次攻击/技能）

1. 计算攻击方 `Hit`（命中率）与防守方 `Dodge`（闪避率）。
2. 计算本次 `FinalHitChance`（最终命中率）。
3. 掷随机数判定是否命中：
   - 命中：继续执行暴击、伤害、减伤、吸血/命中回血等后续流程。
   - 未命中（Miss/Dodge）：本次直接记为 0 伤害，不触发“按造成伤害结算”的收益。

### 6.3 公式定义

| 项目 | 公式 / 规则 |
|------|-------------|
| 最终命中率 | `FinalHitChance = clamp(Hit - Dodge + LevelAdjust, 60, 99)` |
| 未命中率 | `MissChance = 100 - FinalHitChance` |
| 判定 | `rng(0~100) < FinalHitChance` 视为命中，否则未命中 |
| 等级修正 | `LevelAdjust = clamp((AttackerLevel - DefenderLevel) * 0.5, -8, +8)` |

- `clamp(min=60, max=99)`：保证命中与闪避都不会“绝对化”（永不 miss 或永远 miss）。
- `LevelAdjust`：采用推荐版固定参数，每高 1 级 `+0.5%`，每低 1 级 `-0.5%`，总修正封顶 `±8%`。

### 6.3.1 参数定版（推荐版）

| 参数 | 取值 |
|------|------|
| 最低命中下限 `HitFloor` | 60% |
| 最高命中上限 `HitCeil` | 99% |
| 每级差修正 | 0.5%/级 |
| 等级修正封顶 | ±8% |
| 同级修正 | 0% |

> 本表为当前版本定版口径；若后续平衡调整，以本章参数为唯一更新入口，并同步影响装备词缀（`hitPct` / `dodgePct`）的收益评估。

### 6.4 与其他系统的关系

- **暴击**：仅在命中后判定暴击。
- **伤害减免**：仅在命中后进入护甲/抗性计算。
- **武器词缀触发**：生命偷取、命中回血、法力回流等仅在命中且造成有效伤害后结算。
- **仇恨**：未命中不产生伤害仇恨（与 [12-threat.md](./12-threat.md) 一致）。

### 6.5 文档职责边界

- 本章负责“战斗中如何判定命中/闪避”。
- `Hit/Dodge` 的属性来源与面板公式见 [04-classes-attributes.md](./04-classes-attributes.md)。
- 装备词缀只声明“可提供 Hit/Dodge 加成”，不在装备文档重复战斗判定细节。

## 七、战后数据统计衔接

- **触发点（战斗）**：单场战斗**胜负与奖励结算完成后**（或每场行动结束时，与实现一致），将本场**战斗行动步**及战斗内可观测事件汇总并合并进玩家累计统计（伤害与治疗、按英雄与技能拆分、实际暴击/命中/闪避等）。
- **触发点（休息）**：**战后休息**每推进**一步**（或该场休息完成时按总步数一次记入，与实现一致），将**休息步**并入 [13-player-statistics.md](./13-player-statistics.md) 的**累计休息步数**。
- **详细设计**（唯一步数口径、战斗行动步与休息步、探索步数、展示倍率 7.4、**表现层步间毫秒一致 7.5**、累计与清零、护盾吸收计入「受到的伤害」、图表与表）：见 [13-player-statistics.md](./13-player-statistics.md) 1.3–1.6、七。
- **说明**：日志 **[Rx] 回合**为叙事标号；**统计不以战斗回合数为分母**，仅用**步**。**金币/经验效率**用**探索步数**；**战斗强度类**用**战斗行动步**；主界面可对每步数值做 **×10 / ×100** 展示（见该文档 7.4）；**不使用**现实时间作分母。
