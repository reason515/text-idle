# 玩家数据统计与分析系统

## 一、目标与定位

### 1.1 设计目标

为玩家提供**可累计、可清零、可下钻**的战斗与收益数据分析能力，支撑「战术与配装配置 → 多场战斗验证 → 根据数据迭代」的**正向循环**。展示上以**图表**为主（必要时辅以悬停提示、汇总文案等），数值含义清晰。**全部统计派生均以「步」为唯一分母口径**（见 1.3–1.6）：**战斗行动步**（单位每次行动 +1）与**休息步**（战后恢复每 tick +1）；**不以战斗回合数作任何比率的分母**。日志中的 **[Rx] 回合**仅用于叙事与时间轴，**不参与**统计计算。若每步数值过小，界面可采用 **每 10 步 / 每 100 步** 等展示倍率（见 7.4），底层仍以步累计。

### 1.2 与核心循环的关系

| 环节 | 本系统作用 |
|------|------------|
| 布局 (Strategize) | 用历史统计验证技能优先级、目标规则是否合理 |
| 放置 (Engage) | 每场战斗结束后自动写入统计，无需玩家额外操作 |
| 分析 (Analyze) | 小队与英雄多维指标、技能占比、实际命中/暴击/闪避等 |
| 优化 (Optimize) | 直观反馈配装与战术调整的效果 |

### 1.3 唯一口径：步数（无「按回合」统计）

- **原则**：所有可比较的速率、效率、人均指标，分母一律为**步**（战斗行动步、休息步或二者之和），**不存在**「按战斗回合统计」的派生指标（例如不以「每场战斗回合数」为分母的伤害/金币比率）。
- 战斗引擎仍用 **回合** 决定出手顺序（见 [03-combat.md](./03-combat.md)）；**[Rx] 仅出现在日志叙事中**，不进入本系统的累计分母。

### 1.4 战斗行动步

- **一步（战斗侧）** = 某个参战单位（英雄或怪物）完成**一次行动结算**（普攻、技能、待机或其它引擎计为单次行动的类型）。
- **累计战斗行动步**：自统计起点起，所有已结束战斗中上述行动次数之和。可在每次行动 **+1** 或每场战斗结算时从战报汇总，须与引擎一致。
- **用途**：战斗内总量类派生（如伤害、治疗、每步 DPS/HPS 等）的分母为 **累计战斗行动步**（不含休息步）。

### 1.5 休息步

- **战后休息**按 [03-combat.md](./03-combat.md) 4.1：离散步推进；每次 `applyRestStep`（或等价）记 **1 休息步**。
- **累计休息步数**：自统计起点起，所有战后休息消耗的休息步之和。

### 1.6 探索步数（收益效率）

- **探索步数** = **累计战斗行动步** + **累计休息步数**。
- **派生（收益）**：`累计金币 ÷ max(1, 探索步数)`、`累计经验 ÷ max(1, 探索步数)`。
- **界面**：主界面展示见 7.0、7.4（可标为「每探索步」或「每 N 步」）。

## 二、统计周期、累计与清零

### 2.1 累计值

- 以下所有指标在**未清零**的前提下均为**自统计起点起的持续累计**（跨多场战斗、跨会话，具体持久化范围见实现：本地进度 / 服务端存档以产品为准）。
- **每场战斗**根据本场结算与战斗过程事件，**增量更新**各项累计量与**累计战斗行动步**；**每场战后休息**每推进一步（或休息完成时一次性）更新**累计休息步数**；并刷新派生展示（见 1.4–1.6）。

### 2.2 统计起点与清零

| 行为 | 效果 |
|------|------|
| **首次使用** | 统计起点为首次启用或首次成功写入的时刻 |
| **玩家主动清零** | 将所有受本系统管理的累计量归零（或等价重置），**统计起点重置为清零完成后的时刻**；**之后的新战斗**从零开始重新累计 |
| **未清零** | 持续累计；收益类分母为**探索步数**（见 1.6）；战斗强度类分母为**累计战斗行动步**（见 1.4） |

- **清零范围**（设计约束）：至少包含累计量、**累计战斗行动步**、**累计休息步数**及本文档所列派生字段；是否在清零时保留「历史快照」为可选增强，不作为 MVP 必需。

## 三、更新时机

| 时机 | 行为 |
|------|------|
| **每场战斗中 / 每场战斗结束** | 将本场**战斗行动步**并入累计（按行动计数或战后汇总，与实现一致） |
| **每场战后休息**（每步或整段完成，与实现一致） | 将本场休息所消耗步数并入**累计休息步数** |

- **离线战斗**：若与在线采用相同战斗结算逻辑，则离线场次同样在对应结算点写入（与 [01-overview.md](./01-overview.md) 中「数据统计与在线一致」一致）。
- **PVP / 特殊战斗**：是否计入全局统计应在实现中明确；建议默认与 PVE 一致分别累计或提供筛选，避免混淆时在 UI 标注。

## 四、全局与小队层指标

### 4.1 收益类（金币、经验）

| 指标 | 说明 |
|------|------|
| **累计金币获得** | 当前统计周期内，战斗与符合规则的其他来源获得的金币总和（与游戏经济规则一致） |
| **累计经验获得** | 当前统计周期内获得的经验总和 |
| **每探索步金币（派生，主界面）** | 累计金币获得 ÷ **探索步数**（见 1.6）；体现含休息在内的真实效率；展示可乘以 10/100 见 7.4 |
| **每探索步经验（派生，主界面）** | 累计经验获得 ÷ **探索步数** |

### 4.2 战斗输出与生存（小队汇总）

以下均为**当前统计周期内累计**，并派生**每战斗行动步**速率（分母为**累计战斗行动步**，见 1.4；**不含**休息步）；展示亦可按 7.4 缩放：

| 维度 | 小队合计 |
|------|----------|
| **造成的伤害** | 我方对敌方造成的有效伤害等（与实现一致，含过量治疗转伤害等若存在需单列或注释） |
| **受到的伤害** | 我方单位因敌方攻击等失去的生存资源；**含被护盾吸收的部分**（见 5.2） |
| **产生的治疗** | 我方对友方造成的有效治疗等 |

- **从小队汇总到英雄**：小队合计应等于（或可核对）各英雄对应分项之和；若存在环境伤害等无归属项，归入「其他」或按规则分摊并在文档/界面说明。

## 五、英雄（角色）层明细

### 5.1 伤害与治疗（四维）

对**每位英雄**分别累计：

| 类别 | 说明 |
|------|------|
| **造成的伤害** | 该英雄对敌人造成的伤害 |
| **受到的伤害** | 该英雄作为目标所承受的伤害；**真言术：盾等吸收量计入「受到的伤害」口径**（即按「若无护盾则会扣血」的数值统计，与仅统计 HP 下降区分时在 UI 用标签说明「含护盾吸收」） |
| **造成的治疗** | 该英雄对友方产生的治疗 |
| **受到的治疗** | 该英雄作为目标获得的治疗 |

### 5.2 护盾与吸收

- **受到的伤害**：统计敌方（或环境）对该英雄造成的**打击强度**，**护盾吸收部分计入**，以便与「裸吃伤害」的体感一致，并反映坦克与护盾战术的价值。
- 若实现上同时展示「实际 HP 损失」子项，可作为表格中的附加列，避免单一数字歧义。

### 5.3 按技能拆分（伤害与治疗）

- **造成的伤害**：按**技能 ID**（含普攻若单独 ID）拆分占比与绝对值；未知或环境来源归入「其他」。
- **造成的治疗**：同样按**技能 ID** 拆分。
- **受到的伤害 / 受到的治疗**：一般以**来源**（技能或攻击类型）拆分；若 MVP 仅支持来源大类，需在后续版本迭代中补全与 5.3 对称的下钻能力。

## 六、战斗有效性指标（实际率）

对**每位英雄**在**当前统计周期**内，基于**可统计的随机检定次数**计算**实际比率**（非面板理论值）：

| 指标 | 建议定义（实现与战斗引擎事件对齐） |
|------|-----------------------------------|
| **实际暴击率** | 实际发生暴击的次数 ÷ 可暴击的攻击/技能结算次数（需与引擎「哪些动作参与暴击检定」一致） |
| **实际命中率** | 命中次数 ÷ 攻击次数（或「未闪避且完成伤害结算」等，与闪避互斥定义一致） |
| **实际躲闪率（被攻击方）** | 被攻击时成功闪避次数 ÷ 敌方对该英雄的攻击次数（若存在必中、特殊无法闪避，分母或分子规则需统一并在说明中写出） |

- **样本量提示**：当攻击次数过少时，界面应提示「样本不足，比率仅供参考」，避免误导。
- 若某职业存在特殊机制（必暴、必中），比率展示可与技能拆分联动说明。

## 七、展示与交互（图表与表）

### 7.0 主界面与详情页分工

| 区域 | 展示内容 |
|------|----------|
| **主界面** | 仅展示当前统计周期内的 **金币/经验效率**（以步为分母；展示倍率见 7.4）；保持顶栏/主战斗区简洁，不堆叠战斗统计图表 |
| **数据统计详情页**（独立 Modal 或全屏子页） | 其余全部：累计值、**战斗行动步 / 休息步** 子计数、小队/英雄伤害与治疗、按技能拆分、实际暴击/命中/躲闪、图表与表、**清零**等 |

- **进入详情**：主界面效率区域**可点击**打开详情页，或提供独立入口按钮（如「统计」），二者实现选其一即可，避免重复入口过多。
- Tooltip：说明分母为 **探索步数**；列出 **累计战斗行动步** 与 **累计休息步**；**不**将「战斗回合数」作为统计口径展示（日志 `[Rx]` 仍在战斗日志中供阅读）。

### 7.1 原则

- **图表**：用于趋势、占比（如各英雄伤害占比、各技能伤害占比、按场次或随**步数/时间轴**展开的趋势若有时序数据）；**置于详情页**，不在主界面展开。
- **表格**：用于精确数值、四维伤害/治疗、技能级下钻、实际率与样本量；**置于详情页**。
- **层次**（详情页内）：默认「小队总览 → 英雄卡片/行 → 技能明细」，减少一屏信息过载。

### 7.2 UI 规范

- 遵循 [11-ui-tokens.md](./11-ui-tokens.md) 与 `ui-design.mdc`：颜色层级、数值高亮、辅助说明区块样式。
- 需要解释「每探索步」「每 N 步展示」「战斗行动步与休息步」「含护盾」「清零影响」时，使用 `tooltip-wrap has-tip` + `tooltip-text`（见 `frontend-tips-tooltips.mdc`），**不用** `title` 承载机制说明。

### 7.3 清零入口

- 提供明确的「清零统计数据」操作，并**二次确认**（避免误触）；清零后提示**统计起点已更新**。

### 7.4 展示倍率（每 1 / 10 / 100 步）

- **底层**：累计量与 **步数** 的存储、校验、清零始终以 **1 步** 为单位，派生公式仍为 `累计量 ÷ 步数`。
- **显示**：若 **每 1 步** 的数值过小、不便阅读，界面可将同一比率按 **每 10 步** 或 **每 100 步** 放大展示，例如：  
  `显示值 = (累计金币 ÷ 探索步数) × N`，其中 `N ∈ {1, 10, 100}`，标签写清 **「每 100 探索步金币」** 等，避免与底层「每步」混淆。
- **默认**：可由设置或自动根据数量级选择 `N`；切换倍率时 Tooltip 重申当前 `N` 与原始每步换算关系。

### 7.5 表现层：战斗步与休息步等时间隔

- **目的**：统计上战斗**行动步**与**休息步**已同质；**主界面观感**上亦应一致，使玩家对「一步」的体感与数据上的**探索步**对齐。
- **规则（仅 UI 表现，不改结算数值）**：**战斗中每一次行动**在战斗日志中Reveal 所依据的**步间等待**，与**战后休息每推进一步**在日志/动画上的**步间等待**，须使用**同一毫秒配置**（单一来源，例如均由 `getCombatLogStepDelayMs()` 供给，或 `COMBAT_PACING_MS.restStepReveal` 与该值显式相等）；不得长期存在「战斗 3000ms、休息 2000ms」等**不一致**默认值。
- **实现位置**：`frontend/src/game/combatPacing.js` 及消费该节奏的主界面战斗循环；详见 [03-combat.md](./03-combat.md) 1.3。
- **E2E**：快速模式下上述间隔仍为 **0ms**（与现有 `e2eFastCombat` 行为一致）。

### 7.6 详情页 Tab 与场次时序（本场叙事）

- **Tab 结构**：数据统计 Modal 内采用 **Tab** 分区；当前包含 **「概览」**（累计步数、收益倍率、清零）、**「场次趋势」**（本场序列图表）与 **「伤害统计」**（小队伤害占比饼图 + 各角色普攻/技能构成饼图）。不与主界面效率区域堆叠。
- **场次趋势含义**：按 **本统计周期内战斗结束顺序**（左旧右新）展示每场 **回合数**（引擎叙事 `[Rx]`，**不作**「每探索步」等效率分母）、**本场获得金币**、**本场获得经验**。
- **图表读法**：横轴为场次序号；纵轴为 **具体数值刻度**，上限取本周期内 **回合 / 金币 / 经验** 三场峰值中的 **最大值**，三条折线共用该刻度对应每场实际值（回合亦为折线，不单用面积图）。金币线宜用虚线并带点标记以便区分。**图例**置于 SVG 顶部；**回合**折线与 **经验** 折线在视觉上应对齐语义色区分（如 `--color-mp` 与 `--color-magic`，避免与强调色混淆）。精确单场数值可在图上 **悬停对应场次** 时以浮动提示展示（不做场次明细表）。
- **伤害统计**：按统计周期累计我方对敌方造成的有效伤害（战斗日志中带 **普攻 / 技能** 行动的命中条目）；**不含**无法在单次行动中归因的持续伤害（DoT）等。每场战斗结束时增量合并至持久化字段 **`damageByHero`**（按英雄 ID 存 `basic`、`skill` 合计，以及 **`skillById`**：按技能 ID 分档的累计技能伤害，用于与合计交叉校验；无 ID 的技能行计入 `__unknown__`）。**小队占比**饼图仍为各英雄总伤害；**各角色构成**饼图为普攻 + **各具体技能**分扇区（旧存档仅有 `basic`/`skill`、无 `skillById` 时，技能侧仍显示为合并的「技能」扇区；若合计大于 `skillById` 分项之和，则差额显示为「其他技能」）。图例默认仅展示颜色与名称；**数值与占比**在用户悬停扇区或对应图例行时以浮动提示展示。
- **持久化与上限**：每场战斗结算写入一条时间线条目（含 `endedAtMs`）；列表长度有上限（与实现 `MAX_BATTLE_TIMELINE_ENTRIES` 一致），超出时丢弃最早场次。**清零统计**时与累计步数、收益、场次趋势列表、`damageByHero` **一并清空**。

## 九、实现与测试提示（非代码）

- **单场战斗内**：引擎以**回合**编排行动顺序；统计仅累计**步**；**收益效率**分母为**探索步数**；**战斗强度类**分母为**战斗行动步**；无「按回合」比率；不用现实时间作分母。
- **单元测试**：累计、清零、**战斗行动步与休息步**、派生「每探索步」与「每战斗行动步」、护盾吸收计入「受到的伤害」、技能拆分求和与总和一致。
- **E2E**：清零流程、主界面入口可达性、关键图表区域存在性（具体用例随 UI 落地补充）。

## 十、版本与迭代

| 阶段 | 内容 |
|------|------|
| **MVP** | 主界面步数基准收益效率（可选 7.4 倍率）+ 详情页累计/清零 + 战斗行动步与休息步 + 小队/英雄核心指标 + 技能伤害/治疗拆分 + 实际率 + 基础图表/表；**无按回合比率类统计**；**7.5** 战斗与休息步间 UI 间隔一致 |
